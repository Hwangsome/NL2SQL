# Commenting Rules

## 目标

本项目默认面向需要长期维护、交接和演示的工程场景。代码不仅要能运行，还要让后来者快速理解其业务意图和执行路径。

## 总原则

- 新增或修改的代码必须带详细注释
- 注释优先解释业务背景、设计原因、边界条件和异常处理
- 注释不能只把代码翻译成自然语言

## 必须写注释的场景

### 模块级

- 每个核心模块应有模块级说明或顶部文档字符串
- 要说明这个模块负责什么、依赖什么、产出什么

### 类与函数级

- 核心类、核心函数必须有文档字符串
- 至少说明：
  - 功能职责
  - 关键输入
  - 返回值
  - 重要副作用
  - 失败条件或约束

### 逻辑块级

以下逻辑必须加行内注释或块注释：

- Agent 节点执行步骤
- 关键词扩展与召回逻辑
- 表、字段、指标过滤逻辑
- 相对时间解析逻辑
- SQL 生成、校验、纠错逻辑
- SSE 事件封装与流式输出逻辑
- 图表类型自动判断逻辑
- 数据生成中的业务分布控制逻辑

## 注释深度要求

### 推荐写法

- 解释为什么当前分支要这样处理
- 解释这个字段在业务上的含义
- 解释为什么这里要做兜底或回退
- 解释为什么某个排序、过滤或映射规则能提升准确性

### 不推荐写法

- `# 给变量赋值`
- `# 循环列表`
- `# 调用函数`

这类注释没有增加信息量，不满足本项目要求。

## Python 文档字符串建议

建议使用结构化风格，至少覆盖核心输入输出。例如：

```python
def build_query_context(query: str, now: date) -> QueryContext:
    """构建问数上下文。

    该函数负责把用户自然语言问题与当前时间环境组合，生成后续
    检索、时间解析和 SQL 生成所需的统一上下文对象。

    Args:
        query: 用户输入的自然语言问句。
        now: 当前业务日期，用于解析“去年”“本季度”等相对时间。

    Returns:
        供 Agent 各节点共享的标准化上下文对象。
    """
```

## 前端代码补充要求

- 虽然前端不是 Python，但新增复杂前端逻辑同样需要详细注释
- 至少对以下内容补充说明：
  - SSE 流解析
  - 对话状态机
  - 图表推断规则
  - 结果格式转换

## 验收标准

在评估一段新代码是否合格时，应满足：

- 不借助作者口头说明，也能理解主流程
- 遇到复杂逻辑时，注释能解释业务意图
- 遇到时间、SQL、检索、回退分支时，注释能说明设计原因
