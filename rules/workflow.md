# Workflow Rules

## 目标

确保本项目在持续迭代时，代码、文档、数据和运行方式保持一致。

## 目录边界

- `data-agent/`
  Python 后端、Agent、数据脚本、测试、Docker 配置。
- `data-agent-fronted/`
  前端界面与交互逻辑。
- `docs/`
  截图、设计说明、补充文档。
- `scripts/`
  仓库级辅助脚本。

## 修改要求

### 修改后端时

- 优先确认是否需要同步更新提示词、配置样例和测试
- 涉及接口变更时，必须同步检查前端联调是否受影响
- 涉及数据结构变更时，必须同步检查初始化脚本和 README

### 修改前端时

- 保持与后端 `SSE` 协议一致
- 不要假设后端返回一次性完整 JSON
- 有展示逻辑变化时，必要时同步更新 README 截图

### 修改脚本时

- 优先保持命令可重复执行
- 脚本失败时，应输出可定位问题的错误信息
- 脚本说明应写清输入、输出和依赖环境

## 验证要求

### Python 相关

- 依赖变更后先执行 `uv sync`
- 后端或脚本变更后，至少执行相关测试或最小验证命令
- 如可行，优先执行：

```bash
uv run pytest
uv run python -m compileall app
```

### 前端相关

- 影响类型安全时，至少执行类型检查
- 影响构建链路时，至少执行构建验证

## 文档同步

以下场景必须同步文档：

- 启动方式发生变化
- 环境变量发生变化
- 接口契约发生变化
- 数据集路径或脚本名称发生变化
- 页面效果显著变化且 README 依赖截图展示

## 提交前自检

- 是否遵循 `Python + uv` 规范
- 是否补充了足够详细的注释
- 是否补了必要测试或最小验证
- 是否同步更新 README / 配置样例 / 截图
- 是否避免引入无必要的新技术栈
