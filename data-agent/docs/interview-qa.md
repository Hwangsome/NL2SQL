# Data Agent 面试问题与标准回答

这份文档专门整理这个项目在面试中高概率会被问到的问题，以及一套相对稳的回答思路。

目标不是写得像背稿，而是帮你做到两件事：

- 你知道面试官为什么会问这个问题
- 你知道怎么回答会显得更像真的做过

---

## 1. 先准备一段标准项目介绍

### 1.1 一分钟版本

“我做的是一个面向业务问数场景的中文 NL2SQL 系统。用户输入自然语言问题后，系统不会直接让大模型写 SQL，而是先抽取关键词，再分别召回字段、字段值和业务指标，把这些结果合并成候选上下文，再筛选出真正相关的表、字段和指标。之后系统会补充时间和数据库方言信息，再生成 SQL。SQL 生成完不会直接执行，而是先做校验，失败就根据报错自动修正，最后执行 SQL 并把结果总结成业务可读的中文结论。前端通过 SSE 把进度、SQL、结果和结论流式展示出来。这个项目的重点不是单点生成能力，而是把 NL2SQL 做成一条更稳定、可解释的工程化链路。” 

### 1.2 三分钟版本

“这个项目解决的是业务人员不会 SQL，但经常需要临时查数的问题。真实业务场景里，用户说的是业务语言，比如‘销售额’‘华东’‘去年’，但数据库里对应的是另一套表结构、字段命名和指标定义。如果直接让大模型一步生成 SQL，很容易猜错字段、表和 join 条件。  

所以我把整个过程拆成了多阶段。第一阶段是关键词抽取，把自然语言转成检索词。第二阶段是多路召回，分别去召回字段、字段值和指标。第三阶段是合并召回结果，把指标依赖字段补回来，把值命中的样例挂到字段上，并补充关键连接字段，整理成适合 LLM 使用的表级上下文。第四阶段再过滤掉噪声，只保留本次问题真正相关的表、字段和指标。接着系统补充当前日期和数据库方言信息，生成 SQL。SQL 生成后会先校验，如果报错，就把原 SQL 和报错信息一起送去修正节点，保证整个过程不是一次失败就结束。最后执行 SQL，并把结果翻译成业务可读结论，通过前端流式展示给用户。  

我觉得这个项目最关键的价值，不是让模型写一条 SQL，而是如何用检索增强、上下文收敛、SQL 校验修正和过程可视化，把一个不稳定的能力做成相对可控的业务系统。”

## 2. 项目级高频问题

## 2.1 这个项目解决的核心业务问题是什么？

### 标准回答

这个项目解决的是业务人员不会 SQL，但又频繁需要自助查数的问题。

它的核心矛盾在于：用户使用的是业务语言，数据库使用的是表、字段、指标定义和具体值。这两者之间存在明显的语义鸿沟。如果系统不能把这种业务表达稳定地映射到数据库结构上，NL2SQL 就会很不可靠。

所以这个项目本质上是在做一层“业务语义到数据库语义”的映射与收敛。

### 更自然的说法

“这个项目核心不是写 SQL，而是帮不会 SQL 的业务同学查数。真正难的是用户说的是‘销售额、华东、去年’，但数据库里不是这种语言，所以系统必须先把业务语义映射成表、字段、指标和值，最后才能稳定生成 SQL。”

---

## 2.2 为什么不直接让大模型从问题生成 SQL？

### 标准回答

因为在真实业务库里，直接生成 SQL 的不确定性太高，主要体现在：

1. 表和字段名不一定和业务表达一致
2. 指标往往不是单列，而是一套口径
3. 用户经常使用相对时间
4. join 条件容易出错

如果不先做检索和上下文收敛，大模型本质上是在猜。

### 更自然的说法

“如果直接让模型裸写 SQL，其实就是在猜。业务里最怕的不是 SQL 报错，而是跑出来了但含义不对。所以我没有直接让模型一步到位，而是先做召回和过滤，把上下文尽量收窄，再让模型写 SQL。”

---

## 2.3 这个项目为什么要拆成多个节点？

### 标准回答

因为这个问题天然不是单步任务，而是多步推理和多步约束的过程。

拆成多个节点的好处有：

1. 每一步职责明确
2. 更容易观察和调试
3. 每一步都能单独加约束
4. 更容易做错误恢复，比如 SQL 修正

### 更自然的说法

“我拆节点不是为了炫技，而是因为这个问题本来就不是一步能做稳的。比如召回、过滤、生成、校验、修正，这几个动作混在一起会很黑盒，拆开之后每一步在解决什么错误就很清楚。”

---

## 2.4 为什么要做三路召回：字段、值、指标？

### 标准回答

因为一个业务问题通常同时包含三类信息：

1. 字段层信息  
   比如品类、地区、日期

2. 值层信息  
   比如华东、黄金会员、手机

3. 指标层信息  
   比如销售额、订单量、客单价

只做某一类召回，都会丢信息，所以这里做了三路互补召回。

### 更自然的说法

“业务问题不会只包含一种信息。像‘去年华东地区销售额最高的品类是什么’，里面既有指标，也有过滤值，还有维度字段。只召回字段肯定不够，所以我把召回拆成字段、值、指标三路。”

---

## 2.5 为什么字段值召回要用 Elasticsearch，而字段和指标召回用 Qdrant？

### 标准回答

字段和值本质上是两种不同检索问题。

字段和指标更适合做语义相似检索，所以适合向量库。  
字段值更像文本值、枚举值、短语值的检索，更适合全文索引。

因此这里选择：

- Qdrant：字段和指标语义召回
- Elasticsearch：字段值检索

### 更自然的说法

“字段和值不是一回事。字段更像语义匹配问题，比如‘品类’和 `category_name` 的映射，这适合向量检索。值召回更像查文本，比如‘华东’到底是不是地区字段上的值，这更适合 ES 这类全文检索。”

---

## 2.6 `merge_retrieved_info` 为什么很关键？

### 标准回答

因为上游召回结果是零散的，不能直接给 SQL 生成节点。

这个节点做了几件关键补齐工作：

1. 指标反补字段
2. 值反补字段并挂回 examples
3. 按表归组字段
4. 补充 key columns
5. 整理成 prompt 友好的结构

没有这一步，后续生成 SQL 的上下文会非常不完整。

### 更自然的说法

“这个节点不是简单 merge，而是在做上下文补齐。比如只召回到销售额这个指标还不够，我得把它依赖的底层字段补回来；只召回到华东这个值也不够，我得把它挂回地区字段。再加上 key columns，后面模型才更有机会把 join 写对。”

---

## 2.7 为什么在生成 SQL 后还要做校验？

### 标准回答

因为生成结果不等于可执行结果。

校验可以帮助系统显式发现：

- 字段不存在
- 表不存在
- SQL 语法错误
- join 错误

同时，校验失败后还能把错误信息用于自动修正。

### 更自然的说法

“我不把模型输出直接当可信 SQL，而是把它当候选 SQL。先校验，失败就修，再执行。这样整条链路的稳定性会比‘生成就跑’高很多。”

---

## 2.8 为什么前端要流式展示执行过程？

### 标准回答

因为这个系统本质上是一个多阶段 agent 流程，过程可解释性很重要。

前端展示进度、SQL、结论和结果，带来的价值有：

1. 用户更信任系统
2. 开发更容易排查问题
3. 用户能更早看到部分输出

### 更自然的说法

“问数系统如果只有最后一句答案，用户不太敢信。尤其是 SQL 场景，用户会想知道你到底查了什么。所以我把节点进度、SQL 和答案都流式展示出来，让整个过程尽量透明。”

---

## 2.9 你在这个项目里的核心贡献是什么？

### 标准回答

可以从四个维度讲：

1. 链路设计  
   把 NL2SQL 拆成检索、过滤、生成、校验、修正、总结等节点

2. 检索设计  
   设计了字段、值、指标三路召回

3. 上下文设计  
   做了召回结果合并、字段补齐、key columns 补齐、表级结构整理

4. 交互设计  
   用 SSE 把过程和结果流式展示出来

### 更自然的说法

“如果说我这个项目里最核心的贡献，我会说是把整个 NL2SQL 做成了一条更工程化的链路，而不是单个 prompt。包括三路召回、上下文收敛、SQL 校验修正闭环，以及前端的流式展示，这几个点是我最想强调的。”

---

## 2.10 这个项目的局限性是什么？

### 标准回答

这个项目当前更适合：

- schema 相对清晰
- 元数据相对完整
- 单轮问数较多

它的局限主要在：

1. 依赖元数据质量
2. 多轮追问能力还可以继续加强
3. 复杂分析场景的评估体系还需要更完整

### 更自然的说法

“这个系统现在更适合结构化比较清晰的数据域。如果元数据本身不完整，比如字段描述很弱、指标定义缺失，那召回和生成效果都会受影响。另外目前更偏单轮问答，多轮追问和复杂分析场景还可以继续增强。”

## 3. 节点级高频问题

## 3.1 `extract_keywords` 这一步价值是什么？

### 标准回答

它负责把自然语言问题初步转换成更适合检索的词，减少噪声词进入后续召回。

它的重点不是完整理解语义，而是给后面三路召回准备初始检索集合。

### 更自然的说法

“它更像检索前处理，不是最终理解层。先把问题里的业务词抽出来，比如销售额、品类、地区这些，再把整句保留下来做兜底。”

---

## 3.2 `recall_column` 为什么还要让 LLM 先扩词？

### 标准回答

因为用户语言和字段命名不一定一致。  
直接拿用户词去查字段，召回面可能不够。

先扩词可以补充：

- 字段别名
- 同义表达
- 更接近数据库字段命名的形式

### 更自然的说法

“用户说的是业务词，不一定是库里的字段名。比如他说‘品类’，库里可能叫 `category_name`。所以我先让模型补一轮更像字段名的表达，再去做向量召回。”

---

## 3.3 `recall_value` 解决的是哪类问题？

### 标准回答

它解决的是“值到字段”的映射问题。

例如：

- 用户说 `华东`
- 系统要知道这大概率落在 `region_name` 上，而不是其他字段

### 更自然的说法

“很多问题的关键不在字段名，而在值。比如华东、黄金会员、手机，这些都更像字段值。值召回就是为了把这些词挂回正确字段。”

---

## 3.4 `recall_metric` 为什么不能省略？

### 标准回答

因为业务问题经常是指标导向的。  
如果不单独识别指标，系统很容易把指标误当字段，或者漏掉指标口径信息。

### 更自然的说法

“用户最常问的通常就是销售额、客单价、订单量这种指标。如果没有指标召回，系统只能从字段角度去猜，口径很容易偏。”

---

## 3.5 `merge_retrieved_info` 里为什么要补 key columns？

### 标准回答

因为 SQL 能不能写对，很大程度取决于 join key 是否在上下文里。

上游更容易召回业务字段，但真正生成 SQL 时还需要：

- 主键
- 外键
- 维度关联字段

### 更自然的说法

“用户不会问 product_id、region_id，但模型写 join 时必须知道这些列。如果不补 key columns，模型很容易凭空猜 join 条件。”

---

## 3.6 `filter_table` 和 `filter_metric` 的意义是什么？

### 标准回答

召回阶段是偏 recall 优先的，会偏宽；  
过滤阶段是偏 precision 优先的，会进一步裁掉噪声。

这两个节点的本质，是把上下文从“可能相关”进一步压缩成“本次问题真正需要”。

### 更自然的说法

“召回阶段宁可多给一点，避免漏。过滤阶段再把噪声拿掉，避免模型看见太多不相关表和指标。”

---

## 3.7 `add_extra_context` 为什么不是小事？

### 标准回答

它解决了两个很容易被忽视但很实际的问题：

1. 相对时间解析
2. 数据库方言约束

如果没有它，模型在“去年”“本季度”以及 MySQL 语法选择上都可能出错。

### 更自然的说法

“这个节点看起来很小，但非常必要。因为用户天然会说去年、本季度这种词，而且模型如果不知道当前是 MySQL，语法也容易漂。”

---

## 3.8 `generate_sql` 为什么要放在过滤之后？

### 标准回答

因为 SQL 生成非常依赖输入上下文质量。  
如果在候选表和候选指标还很宽的时候就生成 SQL，模型更容易走偏。

### 更自然的说法

“生成 SQL 应该放在上下文已经尽量收紧之后。否则模型看到太多候选表和字段，基本就是在噪声里写 SQL。”

---

## 3.9 `correct_sql` 和重新生成 SQL 有什么区别？

### 标准回答

重新生成是从零开始；  
`correct_sql` 是带着：

- 原 SQL
- 报错信息
- 上下文

做定点修复。

通常后者更稳定，也更可控。

### 更自然的说法

“校正 SQL 更像 debugger，不是重新写作。它知道哪里错了，所以修起来比完全重来更稳。”

---

## 3.10 `summarize_answer` 为什么还要保留 fallback？

### 标准回答

因为最终结果不能依赖 LLM 一定成功。

即使总结模型失败，系统也必须能给用户一个可理解的结果说明，所以保留了规则模板 fallback。

### 更自然的说法

“最后一层不能再是纯赌模型了。就算模型总结失败，系统也得能把结果说明白，所以我保留了规则兜底。”

---

## 4. 工程实现相关问题

## 4.1 为什么后端用 SSE？

### 标准回答

因为这个项目需要持续推送：

- 节点进度
- 中间答案
- 最终结果

SSE 在这种单向服务端推送场景下实现简单、语义清晰，比较适合。

### 更自然的说法

“这类问数链路本来就是服务端持续产出内容，所以 SSE 很合适。它不需要像 WebSocket 那样维护双向协议复杂度，但足够支持进度和答案流式返回。”

---

## 4.2 前端怎么处理 SSE chunk 边界？

### 标准回答

前端没有假设一次 `read()` 就是一个完整事件，而是自己维护了 buffer：

1. 新数据先累加到 buffer
2. 按 `\n\n` 切事件块
3. 最后一个不完整块保留到下一轮

这样能避免 chunk 边界导致的 JSON 解析失败。

---

## 4.3 这个项目怎么做健康检查？

### 标准回答

后端提供：

- `GET /health`

前端会周期性探测这个接口，用来判断服务连接状态。

### 更自然的说法

“前端不是盲目假设后端一定在线，而是会定期打 health 接口，决定当前显示 connected 还是 disconnected。”

---

## 5. 追问“如果继续优化，你会做什么”时怎么答

### 标准回答

我会优先做三类优化：

1. 评估体系  
   建立问题集、SQL 正确率、结果正确率、修正成功率等指标

2. 检索增强  
   例如增加 rerank、增强元数据质量、改进召回策略

3. 多轮问数  
   让系统支持上下文继承，比如连续追问

### 更自然的说法

“下一步我会先把评估做扎实，不只是看‘感觉还可以’。然后会继续优化召回和 rerank，最后再补多轮问数能力，因为真实业务里大家很少只问一轮。”

---

## 6. 面试时建议主动强调的点

建议你在介绍时主动强调下面这些，不然面试官很容易把项目理解成“套个模型写 SQL”：

1. 这是一个多阶段收敛系统，不是单 prompt 系统
2. 核心难点是业务语义到数据库语义的映射
3. 三路召回是为了避免信息缺失
4. `merge_retrieved_info` 是上下文补齐的关键
5. SQL 有校验和自动修正闭环
6. 前端强调过程透明，不是只展示最终结果

---

## 7. 最后一套简洁回答模板

如果你面试现场比较紧张，可以直接按下面这个结构回答：

“这个项目是一个面向业务问数的中文 NL2SQL 系统。  
我没有直接让模型从自然语言生成 SQL，因为真实业务库里这样做很不稳定，容易猜错字段、指标和 join。  
所以我把链路拆成了多个节点：先抽关键词，再分别召回字段、字段值和指标；然后把这些结果合并成候选上下文，再过滤成最小必要集合；接着补充时间和数据库上下文，生成 SQL；生成后先校验，失败就自动修正，最后执行并总结结果。  
我觉得这个项目最重要的点不是用了哪些模型，而是我把一个不稳定的能力做成了一条更可控、可校验、可解释的工程链路。”
